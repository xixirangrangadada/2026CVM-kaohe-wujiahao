## 阶段0：环境搭建

### 需求拆解
搭建能采集真实 PMU 数据的 Linux 环境 + GitHub 仓库骨架。

### 迭代与问题排查

**问题1：WSL2 采不到 PMU**
- 判据：盯死 4 个关键事件（cache-misses / L1-dcache-load-misses / LLC-load-misses / dTLB-load-misses）
- 实测：WSL2（Hyper-V）全 `<not supported>`
- 根因（`dmesg` 一锤定音）：`unsupported p6 CPU model 151 no PMU driver, software events only`（Alder Lake 内核无 PMU 驱动 + Hyper-V 不透传 MSR）
- 决策：换裸金属/原生 Linux

**问题2：ARM 云 PMU 事件语法**
- ARM 云（鲲鹏 920）`perf list` 有 `armv8_pmuv3` 事件，但 `perf stat -e armv8_pmuv3_0/ll_cache_miss_rd/` 报 "unknown term"
- 根因：perf 4.19 把 `pmu/符号名/` 的符号名当成 config term 解析（新版 perf 才支持符号名）
- 解决：从 `/sys/bus/event_source/devices/armv8_pmuv3_0/events/` 读 event code，改用 raw 写法
  - `LLC-load-misses` → `armv8_pmuv3_0/event=0x037,name=LLC-load-misses/`（ll_cache_miss_rd）
  - `branch-instructions` → `armv8_pmuv3_0/event=0x021,name=branch-instructions/`（br_retired）

**问题3：stress-ng 仓库无**
- openEuler 仓库无 stress-ng → `git clone` + `make`（0.21.03），5 个 method 全可用

**问题4：GitHub push 网络**
- HTTPS `curl 28` 断连 → 重试成功

### 关键命令
```bash
# PMU 事件 code 查询
cat /sys/bus/event_source/devices/armv8_pmuv3_0/events/ll_cache_miss_rd   # → event=0x037
# stress-ng 编译
git clone --depth 1 https://github.com/ColinIanKing/stress-ng.git && make -j4
```

### 结果
ARM 环境就绪（鲲鹏 920 / TaiShan v110 / paranoid -1），仓库骨架 push 上线。

## 题1① perf stat 五场景

### 需求
5 负载（int64 / matrixprod / read64 / rand-set / queens）perf 采集 + 衍生指标对比。

### AI 编写 `run_perf_stat.sh`
```bash
taskset -c 0 perf stat -e cycles,instructions,cache-references,cache-misses,\
L1-dcache-load-misses,...,armv8_pmuv3_0/event=0x037,name=LLC-load-misses/,\
branch-misses,armv8_pmuv3_0/event=0x021,name=branch-instructions/,... \
-- stress-ng --cpu 1 --cpu-method int64 -t 30s
```

### 发现
1. **`cache-misses` ≡ `L1-dcache-load-misses`**：perf 4.19 在 ARM 把这两个通用名映射到同一 PMU 事件（L1d refill），冗余。
2. **matrixprod 软浮点**（题1② 火焰图证实）：`__multf3`/`__addtf3`（gcc 软浮点库）→ ARM 无硬件四精度浮点。
3. **queens IPC 1.52**：分支失败率 12.19%（N-皇后条件分支难预测）。
4. **rand-set sys time 30%**：内核态 munmap/pagemap 活动。

### 衍生指标对比表（节选）
| 场景 | IPC | LLC Miss% | 分支失败% |
|---|---|---|---|
| int64 | 2.22 | 0.00019% | 0.031% |
| matrixprod | 2.44 | 0.0053% | 0.90% |
| read64 | 2.49 | 0.0020% | 0.011% |
| rand-set | 2.62 | 0.36% | 0.032% |
| queens | 1.52 | 0.00015% | **12.19%** |

## 题1③ Cache Line 微基准

### 需求
AI 辅助写 pointer chasing C 程序，stride 1–256，验证 cache line 对性能影响。

### 迭代与问题排查

**问题1：延迟全 0**
- 现象：扫 9 个 stride 延迟全是 0.00 ns
- 根因：`gcc -O2` 把 `chase` 循环消除了（循环结束后 `p` 未被使用，编译器判定无副作用）
- 解决：循环体加 `asm volatile("" : "+r"(p) :: "memory")` 内存屏障，强制保留 load

**问题2：拐点不如经典预期**
- 16MB 数组在 L3（128MiB）内，L1/L2 line cross 由 L3 兜底，没有"经典陡峭 64B 拐点"
- 主拐点 stride=64（每 line 正好 1 个节点）；最大单步跳变在 8→16（节点密集度减半）

### 发现
1. **火焰图 stride=8 vs 64 无差异**：两者都是 `chase` ~99.85%（纯用户态 load 循环），调用栈几乎相同。
2. **关键发现**：pointer chasing 的 cache miss 由**硬件**处理（不进内核调用栈），所以 cache line 差异体现在**延迟曲线 + PMU 计数**，而非火焰图调用栈——这正是本题必须用「延迟测量 + perf stat」、不能只靠火焰图的原因。
3. **LLC miss rate 反直觉**（stride 小反而高）：stride 小时 nodes 多、随机密集访问整个 16MB，L3 冲突/容量 miss 概率高；stride 大时有效 footprint 小。

### 延迟曲线
| stride(B) | 1/2/4/8 | 16 | 32 | 64 | 128 | 256 |
|---|---|---|---|---|---|---|
| 延迟 ns | ~21 | 32 | 40 | 43.6 | 44 | 46 |

## 方法论沉淀（AI 协助总结）
1. **决策树驱动**：先盯死几个关键事件作判据，一次跑完看结果，不瞎试。
2. **隔离变量**：把"权限"（paranoid）和"平台能力"（PMU 透传）两个维度拆开归因。
3. **读源头**：最终答案在 `dmesg` / `/sys` / `/proc`，不在 perf 的报错文案里。
4. **跨边界脚本走 heredoc**：`ssh arm 'bash -s' <<'REMOTE'`，避免引号/路径转义。
