# 题2 环境验证记录

> 容器化持续 CPU Profiling —— 地基环境验证
> 验证脚本：`verify_base.sh`（同目录，可复现）
> 日期：2026-06-17

---

## 一、验证目的

题2 核心是"容器化持续 Profiling"，最大的技术不确定性在于：**容器内的 perf 能否采集并解析"另一个 namespace 进程"的用户态符号**（考题测试验证要求"火焰图中能看到 stress-ng 的 matrixprod 热点"）。

裸机 perf 在题1 已验证可用，此处只测**容器化带来的新风险**，不重复测题1 已验证项（perf/FlameGraph/stress-ng 本身可用）。

---

## 二、环境搭建

### 1. Docker 安装
- openEuler 20.03 源里 Docker 包名为 `docker-engine`（非 `docker`/`moby-engine`）
- 安装：`dnf install -y docker-engine`（18.09.0，175MB）
- 启动：`systemctl enable --now docker`，存储驱动 overlay2

### 2. 镜像源踩坑
- **Docker Hub 直连超时**（国内网络）：`docker pull openeuler/openeuler:20.03` 报 `registry-1.docker.io ... Timeout`
- 配阿里云加速无效（`registry.cn-hangzhou.aliyuncs.com` 是镜像仓库非加速器）
- **解决：用 DaoCloud 加速站前缀直拉**：`docker pull docker.m.daocloud.io/openeuler/openeuler:20.03`，成功，264MB aarch64
- 打短标签：`docker tag ... openeuler:20.03`

### 3. 磁盘腾挪
- 原仅剩 3.9G，删 3 个冗余压缩包（gcc/kernel/Hyper-tuner 安装包，解压版保留）腾至 5.2G
- 删除前已备份原件到本地，题后还原

## 三、三关卡验证

`verify_base.sh` 把验证拆成三道关卡，由浅入深，任一关失败可定位到具体层级。

| 关卡 | 测试内容 | 方法 | 结果 | 证据 |
|---|---|---|---|---|
| 1 | Docker PMU 是否开放 | 特权容器内 `perf stat -e cycles -a` | ✅ PASS | 跑出 cycles/instructions 计数，IPC 0.76 |
| 2 | 能否采到别的进程 | `--pid=host` 容器 `perf record -a -g` | ✅ PASS | 4892 samples / 1.3MB perf.data |
| 3 | 符号能否解析 | perf script + FlameGraph，统计有名字符号数 | ✅ PASS | 34929 有名 / 15222 unknown，热点含 stress_cpu/stress_run |

### 容器挂载方案（镜像无 perf）
openeuler 镜像不含 perf，验证阶段挂载宿主机 perf：
```
-v /usr/bin/perf:/usr/bin/perf:ro
-v /usr/lib64:/usr/lib64-host:ro     # perf 依赖库，容器内 LD_LIBRARY_PATH=/usr/lib64-host
-v /lib/modules:/lib/modules:ro       # 内核符号（解析内核栈）
-v /sys:/sys
--privileged --pid=host               # PMU 权限 + 共享宿主 PID ns
```

## 四、踩坑与排查（AI 协作排错过程）

### 坑 1：Docker 是否隔离 PMU？
- **疑问**：`--privileged` 是否真能让容器访问 PMU？只读 `/proc/sys/kernel/perf_event_paranoid`=-1 不能证明 perf 真能用（中间隔着 capabilities/seccomp/cgroup）。
- **排查**：关卡1 实测 `perf stat` 在特权容器内能否跑出计数 → 能，确认 `--privileged` 在 docker18.09+kernel4.19 开放 PMU。
- **教训**：实测优先于假设。

### 坑 2：关卡3 栈全是 [unknown]
- **现象**：perf.data 有数据（关卡2过），但 perf script 输出全是 `[unknown]`，仅零星内核栈。
- **根因**：**时序错误**。脚本原让 stress-ng 先起、perf 后起。perf 靠 **MMAP 事件**建立"地址→符号"映射；stress 已 mmap 完，perf 启动错过其 MMAP 事件 → 采到地址却解不出符号。
- **修复**：调时序——perf 先启动采集，3 秒后再起 stress，让 perf 捕获 stress 的 COMM/MMAP 事件。
- **验证**：修复后符号解析成功（34929 个有名符号）。

### 坑 3：判定逻辑写错
- **现象**：关卡3 判定 `grep "matrixprod"` 永远不命中。
- **根因**：`matrixprod` 是 stressor 的**方法名**（命令行参数），不会出现在调用栈里；栈里真实热点是软浮点函数（题1② 见过 `__multf3`/`__addtf3`）。
- **修复**：判定改成"统计能解出名字的用户态符号数（`+0x` 偏移形式）> 阈值"。

## 五、结论（指导后续组件开发）

1. **`--privileged --pid=host` 方案成立** → PMU 权限 + 跨进程采集都通，题2 容器化可继续。
2. **perf 必须先于目标进程启动** → 否则漏 MMAP 事件致栈全 `[unknown]`。常驻采集器（C1）天然满足此条件，但实现时注意启动时序（采集器先起，被采进程后起）。
3. **镜像无 perf** → 验证期挂载宿主机 perf；正式 Dockerfile 里 `dnf install perf`，不挂载。
