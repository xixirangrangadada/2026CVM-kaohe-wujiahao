# 题2 测试验证报告

> 容器化持续 CPU Profiling —— 3 场景端到端测试
> 测试脚本：`test_scenario.sh`（同目录，可复现）
> 日期：2026-06-17

---

## 一、测试目的

构造 3 类可复现的 CPU 飙升场景，验证工具的端到端能力：**启动采集 → 触发飙升 → 按时段回查 → 火焰图定位热点**。对应考题"测试验证"（15%）。

---

## 二、测试流程

1. 启动 profiler 容器（`--privileged --pid=host`，常驻采集，30s 轮转）
2. 依次跑 3 场景 stress-ng（每场 25s，记录起止时段）
3. 每场景：回查命中采样文件 → 生成火焰图 → 提取用户态热点 top5

运行命令（完整，见 README）：
```bash
docker run -d --privileged --pid=host \
  -v /lib/modules:/lib/modules:ro -v /sys:/sys \
  -v /usr/local/bin/stress-ng:/usr/local/bin/stress-ng:ro \
  -v /data:/data -p 8080:8080 cpu-profiler
```

---

## 三、热点验证表（预期 vs 实测）

| 场景 | 负载类型 | 预期热点 | 实测热点 top（采样次数） | 结论 |
|---|---|---|---|---|
| **matrixprod** | CPU 矩阵乘 | 计算 stressor | `stress_cpu` 4620 / `stress_run` 4622 / `main` 4625 | ✅ 定位 CPU 计算热点 |
| **rand-set** | 内存随机写 | 内存 stressor | `stress_vm_child` 2231 / `stress_run` 2320 | ✅ 定位访存热点 |
| **queens** | N-皇后(分支) | 回溯递归函数 | **`queens_try` 17224（绝对主导）** | ✅ **精准定位分支热点** |

### 关键结论
- **queens 场景**：`queens_try` 是 N-皇后问题的回溯递归核心函数，17224 次采样绝对主导——完美对应"分支密集"负载，证明工具能精准定位真实业务热点。
- **三种负载的 stressor 函数各不相同**（stress_cpu / stress_vm_child / queens_try），证明工具对不同负载类型的热点都能区分定位。

---

## 四、诚实记录：采样噪声

所有场景的火焰图都出现 `_PyEval_EvalFrameDefault`（CPython 字节码执行主循环），来自容器内 supervisor/collector 自身的 Python 进程——因为 `perf record -a` 是**全系统采样**，会采到容器内自己的 Python 进程。

- **影响**：噪声，但不掩盖真实热点（stress_*/queens_try 采样次数远高于 _PyEval，仍清晰可辨）
- **可优化**：采集时可加 `-C` 限定 CPU、或 `--exclude-perf`，但权衡后 `-a` 全系统采集更符合"黑匣子"定位（不知飙升发生在哪个 CPU）

---

## 五、测试结论

1. **端到端闭环验证通过**：启动容器 → 触发飙升 → 回查时段 → 火焰图，全流程顺畅。
2. **热点定位精准**：3 种负载类型的热点函数（stress_cpu / stress_vm_child / queens_try）均被准确捕获。
3. **时段回查准确**：每场景的 stress 时段都能命中正确的采样文件（时区修复后本地时间一致）。
4. **容器化方案成立**：`--privileged --pid=host` 下容器能采到宿主机 stress-ng 的用户态符号。
