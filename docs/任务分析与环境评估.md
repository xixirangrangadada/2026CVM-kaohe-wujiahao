## 二、环境评估（ARM 云，已实测）

### 2.1 硬件（主机 kp-022）

- **CPU**：鲲鹏 920（HiSilicon，implementer 0x48 / part 0xd01），**TaiShan v110** 微架构，aarch64，4 核 @2.6GHz，BogoMIPS 200.00
- **缓存**：L1d/i 256KiB / L2 2MiB / **L3 128MiB**
- **cache line**：L1/L2 = 64B，**L3 = 128B**（题1③ 要标双拐点）
- **NUMA 拓扑**（`numactl --hardware`）：1 node，4 核（0-3）全在 node 0，node 0 内存 6811 MB，距离矩阵 `0→0 = 10`（单插槽，无跨节点延迟）
- **频率策略**（`scaling_governor`）：`performance`（钉最高频，避免动态调频干扰 perf 计数）

> 鲲鹏 920 是华为服务器 ARM 芯片、TaiShan v110 核、大 L3——本身就是好的微架构分析素材。
> NUMA 单 node + performance 调速器 = 测量环境干净（无跨节点延迟、无调频噪声），只需 `taskset -c 0~3` 钉核即可。

### 2.4 PMU 事件（关键，已实测）

**通用名直接可用**：cycles, instructions, cache-references, cache-misses, L1-dcache/icache-load-misses, branch-misses, dTLB/iTLB-load-misses, context-switches, cpu-migrations

**2 个 perf 4.19 不认符号名，用 raw event code 引用**：

| 题目事件 | ARM 原生事件 | code | 写法 |
|---|---|---|---|
| LLC-load-misses | ll_cache_miss_rd | 0x037 | `armv8_pmuv3_0/event=0x037,name=LLC-load-misses/` |
| branch-instructions | br_retired | 0x021 | `armv8_pmuv3_0/event=0x021,name=branch-instructions/` |

> perf 4.19 解析 `pmu/符号名/` 时把符号名当成 raw config term → 报 "unknown term"；必须用 `pmu/event=0xNN/` 或 `rNN` 短形式。事件 code 从 `/sys/bus/event_source/devices/armv8_pmuv3_0/events/<name>` 读。

## 四、决策与路线

**环境定为 ARM 云（kp-022），通过 `ssh arm`（`~/.ssh/config` 别名，ed25519 密钥免密）协作。阶段0 已完成。**

### 环境选型决策树（完整复盘）

**策略**：先本地（零成本）→ 不行再云（有成本）。沿决策树逐候选验证。

#### 候选 1：WSL2（i7-12700K / Alder Lake）
- **为什么先试 WSL2**：本地、简单、零成本；Alder Lake 8P+4E 异构本身是分析素材。
- **AI 预判**：结果可能不太理想（Hyper-V 层 PMU 支持已知是弱项）。
- **实测**：`perf stat -e cycles,instructions` → 全 `<not supported>`。
- **根因**：Hyper-V 不透传 PMU MSR，叠加旧内核无 Alder Lake PMU 驱动（详见附录 A.1/A.1.1）。
- **结论**：❌ 残缺，淘汰。

#### 转折：实验课经验触发"转云"思路
WSL2 失败后，想到**实验课曾用华为提供的 telnet 平台做过 perf 分析**，意识到"云端平台可能具备本地 WSL2 缺失的 PMU 能力"，遂将搜索范围转向云方案。

#### 候选 2：VirtualBox（考虑过，未实测）
- **动机**：想试 Type-2 虚拟机，理论上可调 CPU 直通设置。
- **AI 协作判断**：VirtualBox 主机交互体验差、PMU 透传不开放，与 WSL2/Hyper-V 在 PMU 透传上属同类问题，**大概率同样残缺**。
- **决策**：接受 AI 判断，**不实测直接放弃**，避免在注定失败的方案上浪费时间。
- **结论**：❌ 基于推理放弃（未实测 perf）。

#### 候选 3：ARM 云（鲲鹏 920 / KVM）
- **动机**：真 ARM 服务器硬件、KVM 透传 PMU、openEuler 原生环境。
- **实测**：`perf list` 有 `armv8_pmuv3` 事件、`paranoid=-1`、4 关键事件可用。
- **结论**：✅ 定为执行环境。

#### 决策树总结
```
WSL2（实测残缺）──────────────┐
                              ├─ 本地方案不可行 → 上云
VirtualBox（AI 推理放弃）─────┘
                              ↓
ARM 云（实测可用）✅
```

> **方法论沉淀（双重）**：
>
> **排查方法论**：盯死 4 个关键事件（cache-misses / L1-dcache-load-misses / LLC-load-misses / dTLB-load-misses）作判据，一次跑完看结果；沿"环境栈"（shell → perf → perf_event → 内核 → 虚拟化 → 硬件）逐层下钻定位断裂层；最终答案在 `dmesg` / `/sys` / `/proc`，不在 perf 的报错文案里。
>
> **决策方法论**：(1) **经验迁移触发思路转换**——WSL2 失败后，由"实验课用华为 telnet 平台做过 perf"的经验触发"转云"思路，已有经验是方案搜索的重要线索。(2) **AI 协作做技术推理止损**——VirtualBox 未实测即放弃，基于 AI 对"Type-2 虚拟化 PMU 透传同类问题"的判断，合理利用 AI 推理能力，避免在注定失败的方案上浪费时间。(3) **决策树驱动**——多候选 + 实测/推理结合，系统性排查而非"试一个不行就慌"。

## 六、各小题技术要点（ARM 视角）

### 题1① perf stat 五场景
- 5 负载（stress-ng 0.21.03）：int64→ALU / matrixprod→FPU+SIMD / read64→内存带宽+L3 / rand-set→TLB+CacheMiss / queens→分支预测
- 事件集：通用名 + 2 raw code（见 2.4）
- **每条命令加 `taskset -c 0~3`** 钉死一个核（鲲鹏 920 同构 4 核，避免线程迁移噪声；无 P/E 异构）
- 衍生指标：IPC=ins/cyc、L1/LLC/TLB Miss Rate、分支预测失败率

### 题1② 火焰图
- `perf record -F 99 -g` → `perf script` → `stackcollapse-perf.pl` → `flamegraph.pl`
- 计算密集 = "尖塔"（热点集中）；访存/分支 = "扁平"（热点分散）
- 内核态函数（`__do_page_fault` / `copy_page` / TLB refill）出现要解释成因

### 题1③ Cache Line 微基准 ⚠️ 最易踩坑
- **pointer chasing**（链表节点、节点间距 = stride、沿指针链遍历，绕过硬件预取器）；朴素 `for i: sum += a[i*stride]` 会被预取器抹平拐点
- stride 1/2/4/8/16/32/64/128/256，每 stride 测延迟 + `perf stat` 采 L1/LLC miss（LLC 用 `ll_cache_miss_rd`）
- stride=1 vs stride=64 出火焰图
- **拐点：64B（L1/L2 line）+ 128B（L3 line，鲲鹏 920 特色）**，曲线标双拐点 + 微架构解释

### 题2 容器化 Profiling（暂缓）
- 采集守护 `perf record -F 99 -ag` / 60s 轮转 / 24h 保留自动清理 / 时间段回查 / 一键火焰图
- Dockerfile（aarch64）+ `--privileged` 运行
- 评审为腾讯导师，ARM 镜像声明架构即可
